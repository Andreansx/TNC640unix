Q2 (DECISIVE) — Does PromModule::OnHideStartupPicture self-re-drive activation?
==============================================================================
ANSWER: **NO.** It only clears/tears down the startup-picture state and returns.
A SEPARATE screen-change notification is required afterwards to activate Fred.

PromModule::OnHideStartupPicture (0x6ded0) — full body:
    int OnHideStartupPicture(PromFrame **a1) {
      if ( *((_BYTE*)a1+401) != 0 ) {
        puts("PromModule::OnHideStartupPictureMsg");
        if ( *((_BYTE*)a1+388) != 0 )
          return PromFrame::UnloadStartupPicture(a1[98]);
    LABEL_5:
        *((_BYTE*)a1+388) = PromModule::ConnectPlcServer((PromModule*)a1);   // connect to PLC
        return PromFrame::UnloadStartupPicture(a1[98]);
      }
      if ( *((_BYTE*)a1+388) == 0 ) goto LABEL_5;
      return PromFrame::UnloadStartupPicture(a1[98]);                        // EVERY path ends here
    }
The ONLY things it does: (a) optionally PromModule::ConnectPlcServer (opens Q_PLC_FRONTSTAGE,
builds ClientPlcController/ClientPlcInterfaceImpl — a PLC connection, NOT activation), and
(b) PromFrame::UnloadStartupPicture. It NEVER calls OnScreenChanged / ActivateGroup /
ActivateGroupById / HandleStoredGuiProcess / OnActivateGroupByKey.

PromFrame::UnloadStartupPicture (0x79270) — what it actually does:
  - if a busy/pending counter (this+1094) set -> just marks (this+4381)=1 and returns (defer).
  - if IsStartupPicVisible: PSelectScreenLayout(0,"Screen","Base"); PSelectScreenLayout(1,"Screen","Base")
    (switch screens 0 & 1 back to the Base layout).
  - LogoView::UnloadImage x2 (unload the two logo images).
  - if (this+4319): puts("... Startup phase seems to be over: m_Startup=false"); (this+4344)=0.
  - erase an Rb_tree (this+4384); reset list ptrs.
  - if (m_Startup now 0): PGetForegroundScreen(); PromFrame::SetPlcScreenApi(this, fg);
    (SetPlcScreenApi only pushes the screen-id value out over ClientPlcInterface via a
     PlcApi::tncScreens loop — it does NOT activate a prom group.)
  - if (this+4329): DncConnections GData / PromviewerInterface Connect + call [vtbl+36]
    (a DNC/PLC-connection notification, codes (3,6,18)/(3,6,19) — NOT a group activation).
  => Clears state + updates PLC screen-id + optional DNC notify. No activation re-drive.

Where activation actually comes from (xref proof):
  XrefsTo ActivateGroup(0x7c800): OnActivateGroupByMouse(0x7c976), ActivateGroupById(0x7ca21),
                                  OnScreenChanged(0x7e613, 0x7e648).
  XrefsTo OnScreenChanged(0x7e520): ONLY PromFrame::ScreenChangeObserver::OnNotify(0x7e78e).
  ScreenChangeObserver::OnNotify(0x7e760):

  ScreenChangeObserver::Attach(0x76810): PSession::AddObserver(...)  (subscribes to screen change)
  => The ONLY automatic path to ActivateGroup is:
        <a screen-change/foreground event> -> PSession notify -> ScreenChangeObserver::OnNotify
        -> OnScreenChanged -> (guard now passes since pic hidden) -> ActivateGroup.
CONCLUSION for the emulator: hiding the picture is NECESSARY (it drops the OnScreenChanged
guard) but NOT SUFFICIENT. Prom will NOT activate Fred inside the hide call. After the hide,
a subsequent screen-foreground/screen-change notification must arrive to fire OnScreenChanged.
(This matches the current CLAUDE.md blocker: "persistent screen-foreground arbitration is a
SECOND prom responsibility not yet routed around." UnloadStartupPicture's PSelectScreenLayout is
a LAYOUT swap on screens 0/1, not guaranteed to raise a foreground-change notify, so it is not
a reliable self-trigger.)

Q3 — startup.elf: what gates HideStartupPicture; is it early/unconditional?
==============================================================================
IMPORTANT CORRECTION to the question's premise: FipsNc::NotifyStartupComplete does NOT call
HideStartupPicture. They are two DIFFERENT state machines inside startup.elf:
  * FipsNc  = NC-subsystem coordinator.
  * FipsUI  = MMI/UI coordinator  <-- this is the one that hides the picture.

FipsNc::NotifyStartupComplete (0x507f0) — a transit-queue CALLBACK scheduled by FipsNc::EnterNcOp
(the "State NcOp entered." operational state). Full body:
    FipsIfEvtServer::NotifyStartupComplete(a2[51]);
    FipsIfIpo::NotifyStartupComplete(a2[53]);
    FipsIfPlc::NotifyStartupComplete(a2[54]);
    FipsIfChM::NotifyStartupComplete(a2[55]);
    FipsIfOsci::NotifyStartupComplete(a2[59]);
  => It BROADCASTS "startup complete" to EvtServer, Ipo, Plc, ChM, Osci. It does not touch ProM
     and does not hide the picture. (So "which subsystem NotifyStartupComplete must fire" is the
     wrong axis — none of them hides the picture.)

Who ACTUALLY hides the picture (proven un-fakeably):
  - Full-image raw-pointer scan: the address 0x90ad0 (FipsIfProM::HideStartupPicture) is referenced
    from EXACTLY ONE location: 0x104110 = the FipsIfProM vtable slot (disp 0xB4 from vptr).
  - Indirect-call scan for `call [reg+0xB4]` across all functions: EXACTLY ONE caller ->
    FipsUI::EnterPowerInterrupt @ 0x6d914.
  FipsUI::EnterPowerInterrupt (0x6d8c0):
    EvtMgr::FVerbosePrint(..., "State PowerInterrupt entered.");
    (*(iface_vtbl+0xB4))(iface);          // = FipsIfProM::HideStartupPicture()  -> sends the wire
    (*(iface_vtbl+0xB8))(iface, 19267589); // = SetTitle(0x01260005 "Power interrupted")
    FipsUI::SetSubtitle(a2);
  => The startup picture is hidden EXACTLY when the FipsUI state machine ENTERS the PowerInterrupt
     state (the "Power interrupted" boot screen that the operator acknowledges with CE).

What that transition WAITS ON (message/event driven, NOT a timer/semaphore/config-flag):
  - The NC startup handshake is a MESSAGE cycle:
      FipsNc::CallStartupCycle (0x50790):
        FipsIfDNC::NotifyNcStartupCycle(a2[56], true);
        FipsIfChM::StartupCycle(a2[55]);          // ask the Channel Manager to run the cycle
      FipsNc::InterpretChMStartupCycleAck (0x5c270):
        msg = GetLastReceivedMsg();
        if ( msg && GMessage::IsA(msg, 0xB700E0) && Assign(StUpStartupCycleAck, msg+84) ) OK
        else FVerbosePrint("No or invalid message returned.");
      => it WAITS for the ChM's StUpStartupCycleAck message (GMessage type id 0xB700E0).
  - FipsIfProM handlers that advance the UI machine are message handlers, not polls:
      HandleStUpStartupCycleAck (0x90630), HandleStUpPowerInterruptionAck (0x904f0, sets
      busyAsProgressStep from StUpPowerInterruptionAck body), HandlePlcSrvReply/Notify,
      HandleSTestStartMachine.
  - FipsNc/FipsUI are coupled: FipsNc::EnterNcOp uses AddContentsToWait_StateReached on the UI
    machine and AddTransitQueueEntry_RemoteTransition, i.e. the two machines step forward together,
    gated by PLC/IPO/ChM/self-test acks.
REACHABILITY: NOT early / NOT unconditional. HideStartupPicture is emitted only after the NC
startup cycle progresses far enough for the UI machine to enter PowerInterrupt — i.e. after a
real NC cycle (ChM StUpStartupCycleAck, PLC connect/load, IPO up, self-test). A minimal faithful
boot must deliver those subsystem acks/notifications (chiefly the ChM StUpStartupCycleAck 0xB700E0
and the PLC/IPO startup notifications) so the FipsUI machine reaches PowerInterrupt.
